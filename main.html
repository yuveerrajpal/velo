Existing solutions have a critical flaw.

Smartphone-based fall detection apps work only if your phone survives the crash, stays in range, and has battery. Garmin and Wahoo cycling computers require an active Bluetooth pairing to a smartphone to send any alert - creating a single point of failure at the worst possible moment.

And almost all of them are designed for premium Western markets, not for the realities of Indian cycling: long highway tours, weekend rides on unmarked rural roads, daily commutes through dense traffic where help may be miles away.

VeloSentinel eliminates this dependency entirely. Sensor, processor, cellular radio, and GPS live on a single board - sealed in a weather-resistant enclosure mounted to your bicycle. It works even if your phone is in pieces on the asphalt.

Meet VeloSentinel

One device. Three jobs.

A compact, battery-powered crash detection and emergency response unit, built for Indian cyclists.

01 · DETECT

Detects Real Crashes

A 4-phase finite state machine on the ESP32-S3 reads your IMU 500+ times per second, looking for the exact kinematic signature of a genuine crash: free-fall, impact, tilt, and stillness. Potholes and hard braking trigger zero alerts.

02 · LOCATE

Pinpoints Your Location

The integrated SIM7670G combines 4G LTE with multi-constellation GNSS (GPS + GLONASS + BeiDou). On crash confirmation, it fetches your coordinates and appends a live Google Maps link to the alert SMS - accurate even on remote highways.

03 · ALERT

Alerts Who Matters

Within 3.5 seconds of a confirmed crash, SMS alerts are dispatched to up to 3 pre-configured emergency contacts - family, friends, or 112. All configurable via the onboard Wi-Fi dashboard. No app download required.

How It Works

Four phases.
Every gate must pass.

VeloSentinel uses a sequential 4-phase Finite State Machine. Every phase must be satisfied - in order - before an alert fires.

01
Phase One

Monitoring

The device continuously monitors total acceleration. At rest or riding normally: acc ≈ 9.81 m/s². The FSM waits silently.

acc ≥ 3.0 m/s²
02
Phase Two

Free-Fall

When the bike tips over or goes airborne, the seat-post-mounted device briefly experiences near-weightlessness. acc drops below 3.0 m/s². A 700 ms window opens.

acc < 3.0 m/s²
03
Phase Three

Impact

Within 700 ms, the bike strikes the ground. acc spikes above 25 m/s² (~2.5g). A 2500 ms confirmation window opens.

acc > 25.0 m/s²
04
Phase Four

Validated

Final gates: tilt > 50° (bike on side) AND variance < 2.5 (still). Both must pass. SMS + GPS dispatched.

tilt > 50° & still

Why potholes never trigger an alert.

Road hazards produce a direct impact spike from the IDLE state - with no preceding free-fall phase. Since our FSM requires free-fall BEFORE it will even listen for an impact, all pothole, speed bump, and hard braking events are ignored at the very first gate. Across 4,000 simulated non-crash trials, the algorithm produced zero false positives.

The Algorithm

Transparent.
Tunable.
Explainable.

Unlike black-box ML classifiers - which need labelled crash data that's ethically and practically impossible to collect at scale - VeloSentinel's detection logic is a deterministic finite state machine with four configurable threshold parameters. Every decision is interpretable. Every parameter can be tuned in the field via the onboard Wi-Fi dashboard without reflashing firmware.

// Phase 1: Free-fall detection if (state == IDLE && acc < JERK_THRESHOLD) { state = FREE_FALL; // acc < 3.0 m/s² } // Phase 2: Impact detection if (state == FREE_FALL) { if (elapsed > 700ms) state = IDLE; // timeout if (acc > IMPACT_THRESHOLD) // acc > 25.0 m/s² state = IMPACT_SEEN; } // Phase 3: Stillness + tilt confirmation if (state == IMPACT_SEEN && elapsed > 2500ms) { if (variance < 2.5 && tilt > 50°) CONFIRMED → send SMS + GPS; }
